= 2. How to Start with Mule 4 (Practical Focus)

In this chapter, we will create our first Mule 4 project step-by-step, test it locally, and then deploy it to the cloud, so that we can experience the entire development process from start to finish. The example will be a simple "Hello Mule" API that accepts an HTTP request and returns a text response, but through it we will go through the key steps of integration development.

Creating a new project: After we install Anypoint Studio and launch it (choosing a workspace – the folder where projects will be stored), we create a new Mule project (File > New > Mule Project). We give the project a name, e.g. HelloMule, and select the appropriate Mule runtime version (e.g. Mule 4.4). Studio will open an empty project with a basic structure. In the Package Explorer panel we see the project folder and files (the key one is the mule-project.xml project configuration). In the middle is the Canvas (work surface) where we will graphically build the flow, and on the right side is the Mule Palette with components (connectors, modules). Below is the Properties Editor for configuring component details, and the Console for application logs.

Adding a flow and connectors: Each Mule application consists of one or more flows – sequences of components that process events. We will create a simple flow that triggers on an HTTP call and returns a static response. In the Mule Palette, find the HTTP module (which is usually already enabled by default; if not, it can be added via Add Modules and searching the Exchange repository). Drag the HTTP Listener component from the Palette onto the Canvas. Studio automatically creates a new flow whose source is this HTTP Listener (marked with a blue icon at the start of the flow). The HTTP Listener is the entry point – an HTTP endpoint that listens for HTTP requests on a defined URL and port, and when a request arrives, passes it further down the flow. In the Properties panel for the HTTP Listener, click + to configure the listener. Leave the default Host 0.0.0.0 (which means it listens on all interfaces) and Port 8081. In the Path field enter e.g. /hellomule – this is the path the listener will respond to. With this configuration, we define that our API will accept GET/POST requests at the URL http://localhost:8081/hellomule.

After the HTTP Listener, we add a processor component that will form the response. For this simple example, we will use the Set Payload component (found in the Core module in the Palette). Drag Set Payload onto the Canvas within the Process section (Studio will place it after the Listener). In the Properties for Set Payload, enter the string "Hello Mule" in the Value field (you need to uncheck the Expression flag fx to enter a literal string). This component will simply set the body (payload) of the response to the given text. (Note: in more realistic scenarios, here we would have a series of components calling external systems, transforming data, etc. – all of which we will cover later – but for now we stay with a simple static response to focus on the working mechanics.) Finally, add a Logger component (from the Core module) to log that the flow processed the request, just for tracking purposes. In Logger, we can set the message, e.g. "Processed request on #[attributes.requestPath]" to see the request path.

Local run and test: Save the project (File > Save All). Now right-click on the Canvas and choose Run project HelloMule. In the Console tab watch the log – Mule starts the embedded server and our application. When the message DEPLOYED appears (and the status [✔] Deployed), the application is running locally. It’s time to test: open your favorite REST client (e.g. Postman) and send an HTTP GET request to the URL http://localhost:8081/hellomule. The result should be HTTP 200 OK with the response body "Hello Mule". If we get that – we have successfully developed and run our first Mule application! 🎉 (Note: the 0.0.0.0 host can be replaced with localhost in the URL; the effect is the same.) When finished testing, you can stop the application by right-clicking on the Canvas > Stop Project.

Deploy to CloudHub: Now, let’s "lift" our application to the MuleSoft cloud. We need a MuleSoft Anypoint account (a free trial can be used if this is a test project). In Studio, in the Package Explorer panel, right-click on the HelloMule project > Anypoint Platform > Deploy to CloudHub. Studio will ask for your Anypoint Platform credentials (email and password). Then choose Environment – options include Sandbox (a predefined development environment) or we might have Production environment (for now choose Sandbox). Enter the application name under which it will be available on CloudHub, e.g. hellomule (it must be unique globally – Studio will show a green ✔ if the name is available, or red X if someone else already uses that name). Click Deploy and Studio will start uploading and launching the application in the cloud. After a minute or two, the application should be running on CloudHub. You can verify by logging into Anypoint Platform web interface (the Log in button in Studio deployment wizard or manually via the browser). On the Platform, go to Runtime Manager, select the env (Sandbox), and you should see the hellomule application in the list. Clicking it opens the dashboard with details – including the application URL. CloudHub hosts the application at a URL like http://applicationName.region.cloudhub.io. For applicationName we used hellomule, and region can be e.g. us-e2 (US East) or eu-west-1 (Ireland) depending on the given environment. Studio usually shows the full URL after deploy – for example http://hellomule.us-e2.cloudhub.io/hellomule. You can now invoke this URL from a REST client (or even a browser, since it’s a GET) – you should also get the "Hello Mule" response, only this time the request goes over the internet to our API on CloudHub. Congratulations! We implemented the first Mule flow, tested it locally, and then successfully deployed it to the cloud, completing the full integration development cycle. 🎉
