= 2. Getting Started with Mule 4 (Practical Focus)

In this chapter, we will create our first Mule 4 project step-by-step, test it locally, and then deploy it to the cloud to experience the entire development process from start to finish. The example will be a simple “Hello Mule” API that receives an HTTP request and returns a text response, but through it we will cover the key steps of integration development.

Creating a new project: After installing Anypoint Studio and launching it (choosing a workspace – a folder where projects will be saved), we create a new Mule project (File > New > Mule Project). We name the project, e.g. HelloMule, and select the appropriate Mule runtime version (e.g. Mule 4.4). Studio will open an empty project with the basic structure. In the Package Explorer panel, we see the project folder and files (the key one is the mule-project.xml project configuration). In the center is the Canvas (workspace) where we will graphically build the flow, and on the right side is the Mule Palette with components (connectors, modules). Below that is the Properties Editor for configuring component details, and the Console for application logs.

Adding a flow and connectors: Every Mule application consists of one or more flows – sequences of components that process events. We will create a simple flow that is triggered by an HTTP call and returns a static response. In the Mule Palette, find the HTTP module (usually included by default; if not, add it via Add Modules and searching the Exchange repository). Drag the HTTP Listener component onto the Canvas. Studio automatically creates a new flow whose source is that HTTP Listener (marked with a blue icon at the start of the flow). The HTTP Listener is the entry point – an HTTP endpoint that waits for an HTTP request on a defined URL and port, and when a request arrives, forwards it to the rest of the flow. In the Properties panel for the HTTP Listener, click + to configure the listener. Leave the default Host 0.0.0.0 (which means it listens on all interfaces) and Port 8081. In the Path field, enter e.g. /hellomule – this is the path the listener will respond to. With this configuration, we specify that our API will accept GET/POST requests at the URL http://localhost:8081/hellomule.

After the HTTP Listener, add a processing component that will form the response. For this simple example, we will use the Set Payload component (found in the Core module in the Palette). Drag Set Payload onto the Canvas inside the Process section (Studio places it after the Listener). In the Properties for Set Payload, enter the string "Hello Mule" in the Value field (make sure to uncheck the Expression checkbox fx to enter a literal string). This component will simply set the body (payload) of the response to the given text. (Note: in more realistic scenarios, here we would have a series of components calling external systems, transforming data, etc. – all that we will cover later – but now we stay with a simple static response to focus on the mechanics.) Finally, add a Logger component (from the Core module) to log that the flow processed the request, just for tracking. In Logger, we can set the message, e.g. "Processed request on #[attributes.requestPath]" to see the request path.

Local running and testing: Save the project (File > Save All). Now right-click on the Canvas and select Run project HelloMule. In the Console tab, follow the log – Mule starts the embedded server and our application. When the message DEPLOYED appears (and the status [✔] Deployed), the application is running locally. It’s time to test: open your favorite REST client (e.g., Postman) and send an HTTP GET request to the URL http://localhost:8081/hellomule. The result should be HTTP 200 OK with the response body "Hello Mule". If you get that – you have successfully developed and run your first Mule application! 🎉 (Note: 0.0.0.0 host can be replaced with localhost in the URL; the effect is the same.) When you finish testing, you can stop the application by right-clicking on the Canvas > Stop Project.

Deploy to CloudHub: Now, let’s “lift” our application to the MuleSoft cloud. You need a MuleSoft Anypoint account (a free trial works if this is a test project). In Studio, in the Package Explorer panel, right-click the HelloMule project > Anypoint Platform > Deploy to CloudHub. Studio will prompt you to enter your Anypoint Platform credentials (email and password). Then select Environment – options include e.g. Sandbox (predefined development environment) or you can have a Production environment (for now, choose Sandbox). Enter the application name that will be used on CloudHub, for example hellomule (it must be unique globally – Studio will show green ✔ if the name is free, or red X if someone else uses that name). Click Deploy and Studio will start uploading and launching the application in the cloud. After a minute or two, the application should be running on CloudHub. You can verify by logging into the Anypoint Platform web interface (Log in button in the Studio deployment wizard or manually open the browser). On the Platform, go to Runtime Manager, select the environment (Sandbox), and you should see the hellomule application in the list. Clicking on it opens the dashboard with details – there is also the application URL. CloudHub hosts the application at a URL like http://applicationName.region.cloudhub.io. For applicationName, we used hellomule, and the region can be e.g. us-e2 (US East) or eu-west-1 (Ireland) depending on the environment. Studio usually shows the full URL after deployment – for example http://hellomule.us-e2.cloudhub.io/hellomule. You can now call that URL from a REST client (or even a browser, since it’s GET) – you should also get the "Hello Mule" response, except this time the request goes over the internet to our API on CloudHub. Congratulations! We implemented our first Mule flow, tested it locally, and then successfully deployed it to the cloud, completing the entire integration development cycle. 🎉
