= 2. How to Start with Mule 4 (Practical Focus)

In this chapter, we will create our first Mule 4 project step by step, test it locally, and then deploy it to the cloud, to go through the entire development process from start to finish.

The example we will use is a simple *тАЬHello MuleтАЭ* API that receives an HTTP request and returns a text response. Although the example is simple, it will cover key integration development steps.

== Creating a New Project

After installing Anypoint Studio and launching it, we first select the workspace тАУ the folder where Mule projects will be stored.

A new Mule project is created from the menu:

*File тЖТ New тЖТ Mule Project*

We assign a name to the project, for example `HelloMule`, and select the appropriate Mule runtime version (e.g., Mule 4.4).

After confirmation, Studio creates an empty project with a basic structure. In the *Package Explorer* panel, we see:

* the project folder,
* configuration files (the key one is `mule-project.xml`),
* the `src/main/mule` folder containing Mule flow configurations.

The central part of the screen is the *Canvas* тАУ a workspace for graphical flow creation, while on the right side there is the *Mule Palette* with available connectors and components.
Below is the *Properties Editor* for configuring component details, and also the *Console* for application logs.

== Adding a Flow and Connectors

Each Mule application consists of one or more *flows* тАУ sequences of components that process events.

In this example, we will create a flow that:

* receives an HTTP request,
* returns a static text response.

=== HTTP Listener тАУ Entry Point

In the *Mule Palette*, we find the *HTTP module*. If the module is not already included, it can be added via the *Add Modules* option from the Exchange repository.

From the Palette, drag the *HTTP Listener* component onto the Canvas.
Studio automatically creates a new flow whose *source* is the HTTP Listener (marked with a blue icon at the start of the flow).

The HTTP Listener represents the application entry point тАУ an endpoint that listens for HTTP requests on a defined port and path.

In the *Properties* panel for the HTTP Listener, click the `+` button to create the configuration.

Set the following values:

* *Host*: `0.0.0.0` (listens on all network interfaces),
* *Port*: `8081`,
* *Path*: `/hellomule`.

With this configuration, the endpoint is available at the address:

`http://localhost:8081/hellomule`

=== Set Payload тАУ Forming the Response

After the HTTP Listener, add a processor component that will form the response.

For this example, we use the *Set Payload* component (found in the *Core* module).

Drag the component into the *Process* area on the Canvas, immediately after the HTTP Listener.

In the *Properties* panel:

* enter the text `"Hello Mule"` in the *Value* field,
* disable the *Expression* option (`fx`) to enter a literal value.

This component sets the response payload to a static text.

In real scenarios, this would be where transformations, calls to other systems, and business logic occur, but here we keep the example simple.

=== Logger тАУ Execution Tracking

At the end of the flow, add the *Logger* component (also from the *Core* module).

We use the Logger to record that the request was processed. An example log message can be:

`Processed request on #[attributes.requestPath]`

This way, the logs show the path of the request that passed through the flow.

== Local Running and Testing

After saving the project (*File тЖТ Save All*), run the application locally.

Right-click on the Canvas тЖТ *Run project HelloMule*.

In the *Console* tab, monitor the logs.
When the message appears that the application is *DEPLOYED* and status `[тЬФ] Deployed`, the application is ready for testing.

Testing can be performed using a REST client (e.g., Postman) or directly from a browser by sending an HTTP GET request to:

`http://localhost:8081/hellomule`

If everything is correct, we will receive:

* HTTP status `200 OK`,
* response body: `Hello Mule`.

This means our first Mule application was successfully developed and run locally.

After testing, the application can be stopped via:

Right-click on the Canvas тЖТ *Stop Project*.

== Deploying the Application to CloudHub

The next step is to deploy the application to the MuleSoft cloud environment.

For this, an Anypoint account is required (a trial account is also acceptable).

In the *Package Explorer* panel:

Right-click on the `HelloMule` project тЖТ *Anypoint Platform тЖТ Deploy to CloudHub*.

Studio will prompt for Anypoint Platform credentials, after which we select an environment (e.g., *Sandbox*).

Then enter the application name, e.g., `hellomule`.
The name must be unique globally тАУ Studio will automatically verify this.

After clicking *Deploy*, Studio uploads and launches the application in the cloud.

When the deployment is complete, the application is available at an address of the form:

`http://appName.region.cloudhub.io`

For example:

`http://hellomule.us-e2.cloudhub.io/hellomule`

Accessing this URL yields the same response as locally тАУ `"Hello Mule"` тАУ but now the application runs in the cloud environment.

This completes the full development cycle:

. application development,
. local testing,
. cloud deployment.
